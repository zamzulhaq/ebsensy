BAGIAN B — CARA KERJA SISTEM (SYSTEM FLOW)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[FLOW 1] STARTUP & INISIALISASI
─────────────────────────────────
app.py (baris 1-60):
  1. Load .env → ambil SUPABASE_URL, SUPABASE_KEY, SUPABASE_SERVICE_ROLE_KEY, FLASK_SECRET_KEY
  2. Validasi: jika SUPABASE_URL/KEY kosong → raise Exception (app tidak bisa jalan)
  3. Buat 2 Supabase client:
     - supabase          = client reguler (terikat RLS Supabase)
     - admin_supabase    = client dengan SERVICE_ROLE_KEY (bypass semua RLS) [🔴 KRITIS]
  4. Register semua Blueprint (auth, academic, halaqoh, wali, subscriptions, onboarding, landing)
  5. app.run(debug=True) [🔴 JANGAN PRODUCTION]

[FLOW 2] SISTEM LOGIN — 3 JALUR (app.py baris 169–338)
─────────────────────────────────────────────────────────
JALUR A — Guru via Custom Auth (username + password hash):
  Request POST /login
  → Query tabel 'users' (SaaS users) dengan email
  → Jika tidak ditemukan → Query tabel 'teachers' (legacy)
  → auth/repository.py: TeacherAuthRepository.get_account_by_username()
  → auth/service.py: TeacherAuthService.authenticate() → check_password_hash()
  → Jika valid → set session: user_id, teacher_id, role='guru', school_id, teacher_type, roles[]
  → Load subscription context ke session
  → Redirect ke dashboard sesuai teacher_type (pengajar/halaqoh)

JALUR B — Admin/Wali via Supabase Auth (email + password Supabase):
  Request POST /login
  → supabase.auth.sign_in_with_password({email, password})
  → Ambil profil dari tabel 'profiles' dengan admin_supabase
  → Ambil school_id dari tabel 'user_schools'
  → Jika role='wali' & belum ada school_id → cari dari 'parent_student_relations'
  → Load subscription ke session
  → Redirect dashboard berdasarkan role

JALUR C — Guru Legacy via URL /auth/login-guru (username saja):
  auth/routes.py
  → TeacherAuthRepository → TeacherAuthService.authenticate()
  → Set session manual → redirect ke auth.auto_dashboard

[FLOW 3] OTORISASI & AKSES HALAMAN
────────────────────────────────────
auth/middleware.py:
  @login_required   → cek 'user_id' di session, redirect ke /auth/login jika tidak ada
  @role_required(*roles) → cek session['role'] dan session['roles'], redirect ke auto_dashboard

domains/subscriptions/decorators/subscription_decorators.py:
  @feature_required("nama_fitur") → cek session['features']['nama_fitur']
  → Jika tidak ada → redirect ke /subscriptions/pricing

[FLOW 4] ALUR ABSENSI AKADEMIK (domains/academic/)
────────────────────────────────────────────────────
  1. Guru login → masuk dashboard_pengajar
  2. Pilih kelas → /academic/pilih-kelas
  3. Pilih mata pelajaran → /academic/pilih-subject/<class_id>
  4. Pilih tanggal → /academic/pilih-tanggal/<class_id>/<subject_id>
  5. Input absensi per siswa → /academic/input-absensi/<class_id>/<subject_id>/<date>
     → academic_routes.py → AcademicService.save_attendance() → AcademicRepository.upsert_attendance()
     → Insert/update tabel 'absensi' dengan constraint UNIQUE(student_id, subject_id, date)
  6. Lihat history → /academic/history-absensi/<class_id>

[FLOW 5] ALUR HALAQOH (domains/halaqoh/ + app.py)
──────────────────────────────────────────────────
  1. Guru Halaqoh login → dashboard_halaqoh.html
  2. Lihat daftar santri per halaqoh → /hafalan/halaqoh/<halaqoh_id>
  3. Input hafalan per santri → /hafalan/<student_id> (app.py baris 1436)
     → Validasi form (type, surah, ayat, pages_count, kualitas)
     → Insert ke tabel 'hafalan' via admin_supabase
  4. Weekly report → /admin/weekly-report
  5. Target mingguan → /admin/weekly-targets

[FLOW 6] DASHBOARD WALI MURID (domains/wali/)
──────────────────────────────────────────────
  1. Wali login (via Supabase Auth)
  2. /wali/ → wali_routes.py → WaliService.build_dashboard()
  3. WaliRepository.get_student_relations() → ambil siswa yang terhubung ke wali
  4. Ambil hafalan + absensi siswa tersebut
  5. Build chart kehadiran mingguan (7 hari terakhir)
  6. Render templates/wali/dashboard.html

[FLOW 7] MANAJEMEN SUBSCRIPTIONS
──────────────────────────────────
  1. Di setiap login → app.py load_subscription_to_session() (before_request)
  2. Ambil data dari tabel 'school_subscriptions' JOIN 'subscription_plans'
  3. Cek tanggal expired → jika lewat, downgrade features di session
  4. Decorator @feature_required("analytics") → blokir akses jika plan tidak mendukung
  5. SubscriptionService.check_limit() → cek max_students/max_teachers sebelum tambah data

[FLOW 8] ONBOARDING SEKOLAH BARU (domains/auth/)
─────────────────────────────────────────────────
  /onboarding/register → saas_auth_routes.py → OnboardingService
  → Buat user Supabase Auth
  → Insert ke tabel 'schools', 'profiles', 'user_schools'
  → Assign FREE subscription plan awal

🔴 KRITIS — CEK SEGERA SEBELUM DEPLOY

[R-01] FILE: .env
  MASALAH: SUPABASE_SERVICE_ROLE_KEY dan FLASK_SECRET_KEY ter-commit ke Git.
  DAMPAK: Siapa pun yang akses repo GitHub dapat mengambil alih seluruh database.
  FLASK_SECRET_KEY = "rahsia-flask-001" → sangat lemah, mudah ditebak.
  CEK: Apakah .env ada di .gitignore? Jalankan: git log --all -- .env
  TINDAKAN TIM: Pastikan .env ada di .gitignore, rotate semua key jika sudah push ke repo.

[R-02] FILE: app.py (baris 1626)
  MASALAH: app.run(debug=True, port=5000)
  DAMPAK: Mode debug Flask di production membuka Werkzeug debugger — siapa pun bisa
          eksekusi kode Python arbitrary di server melalui browser.
  CEK: Pastikan ini HANYA ada di development. Production harus gunakan Gunicorn/uWSGI.

[R-03] FILE: app.py (baris 56, 86, + 86 baris lain yang menggunakan admin_supabase)
  MASALAH: admin_supabase (SERVICE_ROLE_KEY) digunakan di banyak route yang seharusnya
           bisa menggunakan client reguler (supabase). Ini bypass semua Row Level Security.
  DAMPAK: Jika ada satu route yang rentan (IDOR, injection), attacker bisa baca/tulis
          SEMUA data dari SEMUA sekolah tanpa batasan.
  CEK: Audit setiap penggunaan admin_supabase — apakah benar-benar perlu bypass RLS?
  FILE TERKAIT: Hampir semua routes di app.py, academic_routes.py, halaqoh_routes.py

[R-04] FILE: app.py (baris 228–260) — LOGIN JALUR B (SaaS users)
  MASALAH: Login menggunakan tabel 'users' dengan plain text password check (tidak ada
           hashing terlihat di segmen ini). Perlu dikonfirmasi apakah password di tabel
           'users' sudah di-hash atau masih plain.
  CEK: Periksa isi tabel 'users' di Supabase dashboard. Pastikan kolom password di-hash.

[R-05] FILE: app.py (baris 340–346) — /register-guru
  MASALAH: Route /register-guru hanya dicek session['role'] == 'guru', tapi tidak ada
           school_id matching. Guru dari sekolah A secara teori bisa mendaftar guru ke
           sekolah B jika school_id dimanipulasi dari form.
  CEK: Validasi bahwa school_id yang dikirim form sama dengan session['school_id'].

[R-06] FILE: app.py, academic_routes.py — IDOR Risk
  MASALAH: Beberapa route mengambil resource berdasarkan ID dari URL parameter tanpa
           memverifikasi kepemilikan school_id.
  CONTOH BERISIKO: /hafalan/<student_id> — apakah selalu filter eq('school_id', school_id)?
  CEK: Setiap query dengan ID dari URL harus selalu disertai .eq('school_id', session['school_id'])
